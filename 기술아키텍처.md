# 어린이 체육시설·강사·팀 매칭 플랫폼 — 기술 아키텍처 설계 (초안)

작성일: 2026-07-21
전제: 요구사항정의서.md 5장(데이터 모델), 6장(MVP 범위), 9장(Founder-led Sales 온보딩 전략), 10장(현장결제), 13장(아동 개인정보보호)을 기반으로 설계

## 0. 설계 전제 — MVP 스코프를 좁히는 핵심 판단

**공급자(시설/강사/팀)는 셀프 서비스 가입 기능을 MVP에 넣지 않는다.**

- 근거: 9장에서 온보딩 전략이 "창업자 직접 영업(Founder-led Sales)"으로 확정되어 있음. 1차 목표도 2개 링크·하키 클럽팀 100% 온보딩이라 규모가 작음(팀 단위 십수 개 수준)
- 따라서 시설/강사/팀/클래스 데이터는 **내부 운영자용 Admin 콘솔에서 직접 입력**하는 구조로 설계 → 공급자용 회원가입/인증/대시보드 개발을 통째로 스킵할 수 있어 MVP 개발 범위가 크게 줄어듦
- 예약 승인/거절도 MVP 단계에서는 운영자가 시설·강사와 전화/카톡으로 확인 후 Admin에서 대행 처리 (6장 "예약 요청 → 승인/자동 확정" 중 **승인 방식으로 시작**, 자동 확정은 데이터 축적 후 도입)
- 2차 확장(공급자 셀프 온보딩, 12.3절 전환 시점 도달 시)에서 공급자 포털을 별도 스프린트로 추가

이 판단에 따라 MVP는 **학부모용 웹앱 + 내부 Admin 콘솔** 2개 화면군으로 구성된다.

## 1. 시스템 구성도 (개념도)

```
┌─────────────────────┐         ┌──────────────────────┐
│   학부모 웹앱 (PWA)   │         │   내부 Admin 콘솔      │
│   Next.js (React)    │         │   Next.js (React)     │
└──────────┬───────────┘         └───────────┬───────────┘
           │  HTTPS / REST                    │ HTTPS / REST (내부망 또는 IP 제한)
           └────────────────┬─────────────────┘
                             │
                   ┌─────────────────┐
                   │   API 서버        │
                   │ Next.js API Route│
                   │  (또는 NestJS)    │
                   └────────┬─────────┘
                             │
        ┌────────────────────┼───────────────────────┐
        │                    │                        │
┌───────────────┐   ┌────────────────┐      ┌──────────────────┐
│ Supabase       │   │ Supabase        │      │ 외부 연동          │
│ Postgres (DB)  │   │ Storage (이미지) │      │ - 카카오맵/로컬 API │
│ + Auth(학부모)  │   │  리뷰사진/자격증  │      │ - 카카오 알림톡 API │
└───────────────┘   └────────────────┘      └──────────────────┘
```

- **학부모용 계정만 로그인 사용자**로 존재 (0절 판단에 따라 공급자 계정 없음)
- API 서버는 학부모 웹앱과 Admin 콘솔이 공유. Admin 경로는 별도 role 체크(운영자 계정)로 분리

## 2. 기술 스택 선정

| 영역 | 선택 | 이유 |
|---|---|---|
| 프론트엔드 | **Next.js (React) + TypeScript**, 모바일 반응형 PWA | 학부모가 주 사용자층 → 모바일 웹 우선. 네이티브 앱은 트랜잭션 검증 후(Phase 2) 판단. Admin 콘솔도 동일 스택으로 재사용해 개발 리소스 절약 |
| 백엔드 | **Next.js API Routes** (또는 별도 NestJS 서버로 초기부터 분리 가능) | 소규모 팀 기준 프론트/백 저장소·배포 파이프라인 단일화로 초기 속도 확보. 트래픽 증가 시 API 레이어만 분리 이관 |
| DB | **Supabase (PostgreSQL)** | 관리형 Postgres + Auth + Storage를 한 번에 제공해 초기 인프라 구축 비용 최소화. 표준 SQL이라 향후 자체 Postgres(RDS 등)로 이관 부담이 낮음 |
| 인증 | **Supabase Auth** (휴대폰 OTP 또는 이메일) — 학부모 전용 | 한국 사용자는 휴대폰 인증 선호. Admin 계정은 별도 이메일+비밀번호, 소수 인원이라 단순 처리 |
| 파일 저장 | **Supabase Storage** | 리뷰 사진, 강사 자격증/프로필 이미지 |
| 지도/위치 | **카카오맵 API (Kakao Local/Map API)** | 국내 서비스 → 주소-좌표 변환, 거리 기반 검색, 지도 시각화에 국내 정확도가 가장 높음 |
| 알림 | **카카오 알림톡(비즈메시지)** + 이메일 백업 | 예약 확정/리마인드/변경 알림. 학부모 도달률이 SMS/이메일보다 높음 |
| 호스팅/배포 | **Vercel** (프론트+API), **Supabase Cloud** (DB) | 서버 운영 인력 없이 관리형 서비스로 MVP 운영 가능. CI/CD는 GitHub Actions → Vercel 자동배포 |

- 이 스택의 공통 원칙: **직접 운영할 서버를 최소화**하고 관리형 서비스로 MVP 속도를 최우선. 10장의 "PG 연동 없이 빠르게 출시"와 동일한 철학을 인프라에도 적용

## 3. 데이터베이스 스키마

5장 엔티티를 관계형으로 구체화. `region_code`는 8장 지역 확장(서울→수도권→전국) 단계에서 지표 집계 기준으로 사용.

```sql
-- 학부모
parents (
  id UUID PK,
  phone VARCHAR UNIQUE NOT NULL,
  email VARCHAR,
  name VARCHAR NOT NULL,
  address VARCHAR,
  region_code VARCHAR,      -- 8장 지역 확장 지표 집계용
  created_at TIMESTAMPTZ
)

-- 자녀 (부모 종속 프로필, 13장: 별도 계정 주체 아님)
children (
  id UUID PK,
  parent_id UUID FK -> parents.id,
  name VARCHAR NOT NULL,
  birth_date DATE NOT NULL,
  gender VARCHAR,
  temperament_tags TEXT[],   -- 성향 태그 (추천 설문 입력값)
  interest_tags TEXT[],
  created_at TIMESTAMPTZ
)

-- 종목
sports (
  id UUID PK,
  name VARCHAR NOT NULL,
  category VARCHAR,          -- 예: 빙상, 구기, 격투 등
  activity_level SMALLINT,   -- 추천 알고리즘용 특성치
  team_or_individual VARCHAR,
  body_part_tags TEXT[]
)

-- 자녀-종목 이력
child_sport_history (
  id UUID PK,
  child_id UUID FK -> children.id,
  sport_id UUID FK -> sports.id,
  started_at DATE,
  ended_at DATE,
  note TEXT
)

-- 체육시설 (Admin에서 운영자가 직접 등록 — 0절 참고)
facilities (
  id UUID PK,
  name VARCHAR NOT NULL,
  address VARCHAR NOT NULL,
  region_code VARCHAR,
  lat DECIMAL, lng DECIMAL,
  phone VARCHAR,
  description TEXT,
  subscription_active BOOLEAN DEFAULT false,  -- 11장 구독료 상태
  created_at TIMESTAMPTZ
)

-- 강사 (Admin 등록)
instructors (
  id UUID PK,
  name VARCHAR NOT NULL,
  facility_id UUID FK -> facilities.id NULL,  -- 시설 소속 선택
  bio TEXT,
  career_years SMALLINT,
  certification_verified BOOLEAN DEFAULT false,
  profile_image_url VARCHAR,
  subscription_active BOOLEAN DEFAULT false,
  created_at TIMESTAMPTZ
)

instructor_sports (              -- 강사-지도종목 M:N
  instructor_id UUID FK -> instructors.id,
  sport_id UUID FK -> sports.id,
  PRIMARY KEY (instructor_id, sport_id)
)

certifications (                 -- 강사 자격 인증 (4.2절)
  id UUID PK,
  instructor_id UUID FK -> instructors.id,
  type VARCHAR,
  issuer VARCHAR,
  document_url VARCHAR,
  verified_at TIMESTAMPTZ
)

-- 팀/클래스 (검색·예약의 핵심 단위, 2장: 학부모가 "등록"하는 대상)
teams_classes (
  id UUID PK,
  name VARCHAR NOT NULL,
  sport_id UUID FK -> sports.id,
  facility_id UUID FK -> facilities.id,
  instructor_id UUID FK -> instructors.id,
  age_min SMALLINT, age_max SMALLINT,
  class_type VARCHAR,        -- individual / group / team
  capacity SMALLINT,
  price INT,
  price_unit VARCHAR,        -- 회당 / 월 / 시즌
  created_at TIMESTAMPTZ
)

-- 반복 시간표 (빈자리 조회의 기준 슬롯)
class_schedules (
  id UUID PK,
  team_class_id UUID FK -> teams_classes.id,
  day_of_week SMALLINT,
  start_time TIME, end_time TIME,
  slot_capacity SMALLINT,
  slot_booked_count SMALLINT DEFAULT 0   -- 동시성 제어용 카운터
)

-- 예약
bookings (
  id UUID PK,
  parent_id UUID FK -> parents.id,
  child_id UUID FK -> children.id,
  team_class_id UUID FK -> teams_classes.id,
  class_schedule_id UUID FK -> class_schedules.id,
  status VARCHAR,             -- requested / confirmed / completed / cancelled
  requested_at TIMESTAMPTZ,
  confirmed_at TIMESTAMPTZ,
  cancelled_at TIMESTAMPTZ,
  cancel_reason TEXT,
  matching_fee_applied BOOLEAN DEFAULT false  -- 11장 매칭수수료 정산 여부 (Phase2)
)

-- 리뷰 (어뷰징 방지: booking_id 필수 → 실제 예약 이력자만 작성)
reviews (
  id UUID PK,
  booking_id UUID FK -> bookings.id UNIQUE NOT NULL,
  parent_id UUID FK -> parents.id,
  target_type VARCHAR,        -- facility / instructor / team_class
  target_id UUID,
  rating SMALLINT,
  content TEXT,
  photo_urls TEXT[],
  created_at TIMESTAMPTZ
)

-- 알림 로그
notifications (
  id UUID PK,
  recipient_phone VARCHAR,
  type VARCHAR,               -- booking_confirmed / reminder / cancelled 등
  payload JSONB,
  sent_at TIMESTAMPTZ,
  channel VARCHAR              -- kakao_alimtalk / email
)

-- 내부 운영자 계정 (Admin 콘솔 전용)
admins (
  id UUID PK,
  email VARCHAR UNIQUE,
  name VARCHAR,
  role VARCHAR
)
```

**인덱스 설계 우선순위**: `teams_classes(sport_id, facility_id)`, `facilities(region_code)`, `class_schedules(team_class_id, day_of_week)` — 검색/필터(4.1절)가 가장 빈번한 쿼리이므로 최우선 복합 인덱스 대상

**동시성 처리**: 예약 신청 시 `class_schedules.slot_booked_count`를 트랜잭션 내에서 조건부 증가(`WHERE slot_booked_count < slot_capacity`)시켜 초과 예약(overbooking)을 DB 레벨에서 방지

## 4. API 설계 (MVP 범위)

| 메서드/경로 | 설명 | 관련 장 |
|---|---|---|
| `POST /auth/otp` `/auth/verify` | 학부모 휴대폰 인증 가입/로그인 | 6장 |
| `GET/POST/PATCH /children` | 자녀 프로필 관리 | 5장 |
| `POST /recommend` | 자녀 특성(나이·성향·체력·관심사) 입력 → 규칙 기반 점수화로 종목 추천 리스트 반환 | 3장(A), 4.1절 |
| `GET /search?sport=&region=&age=&day=&time=&price_min=&price_max=&type=&sort=` | 필터·정렬된 팀/클래스 검색 | 4.1절 |
| `GET /classes/:id` `/facilities/:id` `/instructors/:id` | 상세 페이지 (프로필/시간표/가격/리뷰 포함) | 3장 |
| `GET /reviews?target_type=&target_id=` | 리뷰 목록 | 4.2절 |
| `POST /reviews` | 리뷰 작성 (요청 시 `booking_id` 검증 — status=completed 인지 확인) | 4.2절 |
| `POST /bookings` | 예약 신청 (슬롯 트랜잭션 차감) | 4.3절 |
| `GET /bookings` | 내 예약 목록/상태 조회 | 4.3절 |
| `PATCH /admin/bookings/:id` | 운영자가 승인/거절 대행 처리 → 확정 시 알림 발송 트리거 | 0절, 4.3절 |
| `POST/GET/PATCH /admin/facilities` `/admin/instructors` `/admin/classes` | 운영자용 공급자·클래스 CRUD | 0절 |

- 알림 발송은 `PATCH /admin/bookings/:id`에서 상태가 `confirmed`/`cancelled`로 바뀔 때 카카오 알림톡 API를 서버에서 트리거하는 이벤트 기반 구조로 처리 (별도 큐 없이 MVP는 동기 호출로 충분, 실패 시 재시도 로직만 추가)

## 5. 비기능 요구사항 및 확장 대비

- **개인정보(13장)**: `children` 테이블은 `parent_id`로만 접근 가능하도록 API 레벨 권한 체크. Supabase Row Level Security(RLS)로 "본인 자녀만 조회 가능" 정책을 DB 레벨에서도 강제. 탈퇴 시 자녀 데이터 파기 배치 잡 별도 구현
- **PG 연동 확장(11.1절)**: Phase 1→2 전환 임계값 도달 시 `bookings`에 `payment_id`, `payment_status` 컬럼과 별도 `payments` 테이블만 추가하면 되도록 스키마를 사전에 열어둠 (현재 스키마는 결제 정보 없이도 완결)
- **매칭수수료 정산(11.2절)**: `bookings.matching_fee_applied` 플래그를 미리 넣어두어, PG 연동 후 정산 배치가 이 플래그 기준으로 미정산 건을 조회하도록 설계
- **지역 확장(12.1절)**: `region_code` 기반 집계 쿼리(활성 학부모 수, 매칭 건수)를 KPI 대시보드로 분리 예정. 데이터량이 작은 MVP 단계에서는 파티셔닝 불필요, 추후 필요 시 `region_code` 기준 파티션 테이블로 전환
- **공급자 셀프 온보딩 전환(12.3절)**: 현재 Admin 전용 CRUD API를 그대로 공급자 role 권한 체크만 추가해 재사용 가능하도록 설계(엔드포인트 재설계 불필요, 인증/인가 레이어만 확장)
